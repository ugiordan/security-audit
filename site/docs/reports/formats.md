# Report Formats

The pipeline generates seven report formats from the same triaged findings data. Each format targets a different audience and use case.

## Overview

| Format | File | Audience | Features |
|---|---|---|---|
| **Full HTML** | `security-report.html` | Security engineers | Self-contained, filterable, code snippets, GitHub links |
| **Must-fix HTML** | `must-fix-report.html` | Engineering leads | Critical + high severity only, action-oriented |
| **MkDocs site** | `report-site/` | Team sharing | Multi-page, searchable, hosted or local |
| **Full Word** | `security-report.docx` | Management / compliance | Formal document format, printable |
| **Must-fix Word** | `must-fix-report.docx` | Management | Critical + high only, executive-friendly |
| **Executive MD** | `executive-report.md` | Quick reference | Full findings in markdown |
| **Must-fix MD** | `must-fix-report.md` | Triage meetings | Critical + high in markdown |

## Full HTML report (`security-report.html`)

The primary report format. A single self-contained HTML file with all CSS and JavaScript inlined (no external dependencies). Drop it anywhere and open in a browser.

### Features

- **Stat cards**: Total findings, critical/high/medium/low counts
- **Donut chart**: Visual severity distribution
- **Filter chips**: Click to filter by severity, source tool, or triage status (corroborated/AI-only/SAST-only)
- **Finding cards**: Color-coded severity border, triage badge, source chip, file:line GitHub permalink
- **Code snippets**: Syntax-highlighted vulnerable code
- **Collapsible sections**: Dependency CVEs grouped and collapsed by default
- **CONFIDENTIAL banner**: Red banner at top and bottom

### Card anatomy

See [Card Format](card-format.md) for detailed card structure.

## Must-fix HTML report (`must-fix-report.html`)

Same card-based layout as the full HTML report but filtered to critical and high severity findings only. Intended for engineering leads who need to see what must be fixed before a release.

Generated by:

```bash
python3 report_mustfix.py <scan-dir> --html
```

## MkDocs site (`report-site/`)

A multi-page static site generated by `report_html.py`. Useful for hosting on an internal web server or GitHub Pages. Includes:

- Index page with summary statistics
- Per-severity pages
- Per-category pages
- Individual finding detail pages

Serve locally:

```bash
cd output/<repo>/<timestamp>/report-site
python3 -m http.server 8000
```

## Word documents (`.docx`)

Professional document format for sharing with management, compliance teams, or external auditors.

### Full report (`security-report.docx`)

Generated by `report_docx.py`. Includes:

- Title page with repo name, scan date, and finding counts
- Executive summary with severity breakdown
- Detailed findings grouped by severity
- Each finding includes: title, severity, file path, description, code snippet, and remediation

### Must-fix report (`must-fix-report.docx`)

Same structure but filtered to critical and high severity only:

```bash
python3 report_docx.py <scan-dir> --must-fix
```

## Markdown reports

### Executive report (`executive-report.md`)

Full findings in markdown format. Generated by `report.py`. Suitable for pasting into Confluence, GitHub issues, or Slack.

### Must-fix report (`must-fix-report.md`)

Critical and high findings only. Generated by `report_mustfix.py` (without `--html`).

## Regenerating reports

To regenerate reports from existing scan data without re-running SAST or AI:

```bash
python3 pipeline.py org/repo --reports-only
# Or with a specific scan directory:
python3 pipeline.py org/repo --reports-only --scan-dir output/repo/2026-06-04-142201
```

This re-runs normalize, dedup, triage, and all report generators using the existing raw data.
